# P6 — Autonomous A2A / Client Credentials

**Status:** ✅ Built  
**Auth flow:** No user → Client Credentials (private_key_jwt) → HR + Finance MCP → Slack  
**Right panel:** Mission

---

## The Story

*"This agent runs every night. No one pressed a button. It woke up, used its own identity to check org data, and posted the report to Slack."*

P6 demonstrates a **fully autonomous agent** operating with its own machine identity — no user delegated authority, no human in the loop. The agent uses the OAuth 2.0 Client Credentials grant with JWT private-key authentication (`private_key_jwt`) to acquire resource-scoped tokens, queries HR and Finance data, and posts analysis to Slack as a bot.

The **RUN button** in the Mission panel simulates what in production would be a cron trigger or an event-driven pipeline.

---

## Client Credentials Flow

```
POST https://{OKTA_DOMAIN}/oauth2/{HR_AUTHZ_SERVER_ID}/v1/token

grant_type             = client_credentials
scope                  = hr:read
client_assertion_type  = urn:ietf:params:oauth:client-assertion-type:jwt-bearer
client_assertion       = {signed JWT — see below}
```

Same call against `FINANCE_AUTHZ_SERVER_ID` for `finance:read`. Both requests run in parallel at the start of each mission.

---

## Client Assertion JWT

Signed with the agent's RS256 private key (`P6_OKTA_PRIVATE_KEY`):

```json
{
  "alg": "RS256",
  "kid": "{kid from JWK}",
  "iss": "{P6_OKTA_AI_AGENT_ID}",
  "sub": "{P6_OKTA_AI_AGENT_ID}",
  "aud": "{token endpoint URL}",
  "iat": now,
  "exp": now + 300,
  "jti": "{random UUID}"
}
```

> Tokens are not cached across requests — CC tokens are cheap to issue and always fresh per mission run.

### Mission 1 — Org Pulse Report

**What it does:** Lists all employees, groups them by role and department, composes an org health summary, and posts it to Slack.

**Schedule label (UI):** "Runs nightly at midnight"

**Tools used:**

- Slack: `POST /chat.postMessage` with `SLACK_BOT_TOKEN`

## End-to-End Flow

```
1.  User clicks RUN on a mission card in the console
2.  Console POSTs { message: missionPrompt } to p6-agent /chat
    (no Authorization header — P6 has no user token)
3.  Agent acquires two CC tokens in parallel:
      HR token  ← POST /oauth2/{HR_AUTHZ_SERVER_ID}/v1/token  scope=hr:read
      Finance token ← POST /oauth2/{FINANCE_AUTHZ_SERVER_ID}/v1/token  scope=finance:read
4.  LLM agentic loop:
      → MCP tool calls with CC-issued Bearer tokens
      → HR/Finance servers validate via JWKS; emit token-validated events (sub=P6_OKTA_AI_AGENT_ID)
5.  Agent composes analysis
6.  Agent posts to Slack using SLACK_BOT_TOKEN directly
      (no STS needed — bot acts as itself)
7.  Agent streams final summary back to console
```

## P3 vs P6 — Key Contrast

| Aspect | P3 (XAA) | P6 (Client Credentials) |
|--------|----------|--------------------------|
| User involved | Yes — user logs in, delegates identity | No — agent acts as itself |
| Token `sub` | User email | Agent ID (`P6_OKTA_AI_AGENT_ID`) |
| Token `act` | Agent ID | Not present |
| Grant type | `jwt-bearer` (step 2) | `client_credentials` |
| Auth entry point | User-facing PKCE login | No login — automated |

Running both side-by-side makes the human-delegation concept immediately visible: in P3 the resource servers see the user's identity; in P6 they see only the agent.

## Dual-Entity Architecture

Okta AI Agents (Workload Principals, `wlp_xxx`) must be linked to a **web app** client — not a service app. But the CC grant requires a service-app or confidential client. P6 therefore maintains two entities:

| Entity | Purpose | Env var |
|--------|---------|---------|
| CC service app | Executes `client_credentials` grant on HR/Finance auth servers | `P6_OKTA_CLIENT_ID` (internal, may equal `P6_OKTA_AI_AGENT_ID`) |
| Placeholder web app (`okta-demo-p6-wlp-placeholder`) | Required to satisfy Okta's WLP-must-link-to-web-app constraint | Not exposed in `.env` |

In practice `P6_OKTA_AI_AGENT_ID` is used as both the `wlp_xxx` AI Agent ID and the `iss`/`sub` in the client assertion. The placeholder web app is created only to satisfy the registration requirement and is never used for real OAuth flows.

## RFC 8693 Token Exchange — Mission 1 Original Design (Blocked)

The original Mission 1 design called for an A2A delegation chain: an orchestrator acquires a minimal CC token, spawns worker agents, and each worker exchanges the orchestrator token for a resource-scoped token via RFC 8693 token exchange — with `act` claims showing the full delegation chain.

**Why it's blocked — Okta's restriction matrix:**

| Subject token source | Requesting client | Result |
|----------------------|-------------------|--------|
| Web app (auth_code client) | Any | ❌ "Token Exchange from web app isn't supported" |
| Service app (CC client) | Web app (incl. AI Agent WLP) | ❌ "Token Exchange from service app to web app isn't supported" |
| Service app (CC client) | Service app (CC client) | ✅ Works |

**Root cause:** Okta AI Agents (`wlp_xxx`) are hardwired to web app clients. The token exchange policy rejects requests from web-app-associated clients when the subject token came from a service app. There is no way to give a `wlp_xxx` AI Agent a service app identity.

**What would unblock it:** Okta would need to allow AI Agents to be associated with service app (CC) clients, or introduce a new WLP type that can participate in service-app token exchange.

**Current state:** Mission 1 card is shown in the UI. The agent runs a standard CC flow instead of the RFC 8693 chain. The original delegation chain code is preserved (commented out) in the agent source for reference.

## Slack Integration

P6 posts to Slack using `SLACK_BOT_TOKEN` directly — no Okta STS exchange needed. The bot acts as itself (not on behalf of any user). Required scopes at Slack app installation:

- `chat:write`
- `chat:write.public`
- `channels:read`

```bash
SLACK_BOT_TOKEN=xoxb-...
SLACK_DEFAULT_CHANNEL=general   # channel name without #
```

If `SLACK_BOT_TOKEN` is blank, P6 skips the Slack post and returns the report as plain text in the chat response.

## Key Env Vars

```bash
P6_OKTA_AI_AGENT_ID=          # wlp_xxx Workload Principal ID (JWT iss/sub, also CC client_id)
P6_OKTA_PRIVATE_KEY=          # JWK JSON string (single line) — signs CC assertions

# Shared MCP resource server vars (same as P3)
HR_AUTHZ_SERVER_ID=           # Okta auth server ID for HR
HR_RESOURCE_AUDIENCE=         # Token audience HR server expects (e.g. api:hr)
FINANCE_AUTHZ_SERVER_ID=      # Okta auth server ID for Finance
FINANCE_RESOURCE_AUDIENCE=    # Token audience Finance server expects (e.g. api:finance)

# Slack
SLACK_BOT_TOKEN=              # xoxb-... bot token
SLACK_DEFAULT_CHANNEL=        # channel name without # (e.g. general, demo-reports)
```

## Okta Prerequisites

All provisioned automatically by `node scripts/setup.js`:

- **AI Agent (Workload Principal)** — `P6_OKTA_AI_AGENT_ID`; RSA public key registered
- **Client Credentials grants** on HR and Finance auth servers for the P6 agent client
- **Placeholder web app** (`okta-demo-p6-wlp-placeholder`) — created only to satisfy WLP linkage

**Slack bot token** — manual:
1. Create a Slack app at [api.slack.com/apps](https://api.slack.com/apps)
2. Add Bot Token Scopes: `chat:write`, `chat:write.public`, `channels:read`
3. Install the app in your workspace
4. Copy the `xoxb-...` Bot User OAuth Token
5. Set `SLACK_BOT_TOKEN=xoxb-...` in `.env`
