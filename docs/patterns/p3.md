# P3 — 1st Party XAA Native

**Status:** ✅ Built  
**Auth flow:** User login (PKCE) → id_token → XAA two-step → per-tool scoped tokens → HR + Finance MCP  
**Right panel:** Chat

---

## The Story

*"You log into the company portal and ask the AI assistant to pull up Sarah's org chart position and check the Q3 budget for Engineering. The agent doesn't have standing access to HR or Finance — it goes and gets exactly the permissions it needs for each tool call, carrying your identity the whole way. The resource servers can see it was YOU who authorized this, and the agent acting on your behalf."*

P3 demonstrates **first-party user-delegated identity via Okta XAA (Extended Agent Authorization)**. The agent never holds the user's password or long-lived credentials — it holds an `id_token`, performs a two-step exchange with Okta to obtain narrow-scoped resource tokens, and discards them after use.

---

## XAA Protocol (ID-JAG)

XAA is Okta's two-step token exchange protocol for user-delegated agent identity:

### Step 1 — id_token → ID-JAG

Exchange the user's `id_token` for an **ID-JAG** (an intermediate token embedding user identity + requested scopes, signed for a specific downstream auth server as the audience).

**Endpoint:** `POST https://{OKTA_DOMAIN}/oauth2/v1/token` *(org-level token server)*

```
grant_type            = urn:ietf:params:oauth:grant-type:token-exchange
requested_token_type  = urn:ietf:params:oauth:token-type:id-jag
subject_token         = {user's id_token}
subject_token_type    = urn:ietf:params:oauth:token-type:id_token
audience              = https://{OKTA_DOMAIN}/oauth2/{HR_AUTHZ_SERVER_ID}
                        ← URL of the target auth server (not the org server)
scope                 = hr:read
                        ← REQUIRED in Step 1; embedded into the ID-JAG
client_assertion_type = urn:ietf:params:oauth:client-assertion-type:jwt-bearer
client_assertion      = {signed JWT — see below}
```

Returns: an `access_token` that IS the ID-JAG.

> The `audience` must be the URL of whichever auth server will execute Step 2 — e.g. `https://{OKTA_DOMAIN}/oauth2/{HR_AUTHZ_SERVER_ID}` for HR, or the Finance equivalent. The `default` auth server is not required.

### Step 2 — ID-JAG → resource access token

Exchange the ID-JAG for a resource access token at the target auth server.

**Endpoint:** `POST https://{OKTA_DOMAIN}/oauth2/{HR_AUTHZ_SERVER_ID}/v1/token` *(HR/Finance auth server)*

```
grant_type            = urn:ietf:params:oauth:grant-type:jwt-bearer
                        ← NOT token-exchange
assertion             = {ID-JAG from Step 1}
client_assertion_type = urn:ietf:params:oauth:client-assertion-type:jwt-bearer
client_assertion      = {signed JWT — see below}
```

> **Critical:** Do NOT include `scope` in Step 2. Okta returns `invalid_request: The id-jag request must not include a 'scope' parameter`.

Returns: a resource `access_token` with scopes from Step 1, audience matching `HR_RESOURCE_AUDIENCE` or `FINANCE_RESOURCE_AUDIENCE`.

### Client Assertion JWT (both steps)

Signed with the agent's RS256 private key (`P3_OKTA_PRIVATE_KEY`):

```json
{
  "alg": "RS256",
  "kid": "{kid from JWK}",
  "iss": "{P3_OKTA_AI_AGENT_ID}",
  "sub": "{P3_OKTA_AI_AGENT_ID}",
  "aud": "{token endpoint URL for this step}",
  "iat": now,
  "exp": now + 300,
  "jti": "{random UUID}"
}
```

## Per-Tool Scope Design

Each MCP tool maps to exactly the scopes it requires. The agent requests only those scopes in the XAA Step 1 call. Tokens are cached within a request by `(server, scopeKey)` to avoid redundant XAA round-trips.

| Tool | Server | Scope |
|------|--------|-------|
| `list_employees`, `get_employee`, `get_org_chart`, `list_departments`, `search_employees` | HR | `hr:read` |
| `update_employee_title` | HR | `hr:write` |
| `get_budget`, `list_invoices`, `get_expense_report`, `list_cost_centers`, `get_invoice` | Finance | `finance:read` |

At startup the agent fetches `scopes_supported` from each auth server's OIDC well-known endpoint and warns if any scope in the map is missing from the server config.

## End-to-End Flow

```
1.  User clicks "Login with Okta" on /patterns/p3
    → /api/auth/start/p3 → Okta PKCE (P3_OKTA_CLIENT_ID, openid profile email)
2.  Okta → /api/auth/callback/p3
    → console exchanges code for tokens using P3_OKTA_CLIENT_SECRET
    → id_token stored in HttpOnly cookie p3_id_token
3.  Page reloads → server reads cookie → passes userToken to ChatPanel
4.  User sends a chat message
    → console sends Authorization: Bearer {id_token} to p3-agent
5.  Agent pre-fetches tokens for HR read + Finance read via XAA (cached for this request)
6.  Agent discovers tools from both MCP servers (one-shot connect → listTools → disconnect)
7.  LLM decides which tool to call
8.  For each tool call:
    a. Agent looks up (server, scope) in cache; if miss → runs XAA for that scope pair
    b. Creates one-shot MCP client with Bearer token
    c. Calls tool → closes connection
9.  HR/Finance MCP servers validate token via JWKS
    → emit token-validated event with sub (user) + act (agent) claims
10. Agent streams response to console
```

---

## Token Anatomy (Step 2 result)

```
iss  = HR or Finance auth server
sub  = user email/ID (delegated identity — the human who authorized this)
act  = { sub: P3_OKTA_AI_AGENT_ID }  ← the agent acting on the user's behalf
aud  = api:hr  or  api:finance
scp  = hr:read  (or finance:read, hr:write, etc.)
```

The `sub` + `act` claim pair is the key demo talking point: resource servers can see BOTH identities.

## Key Env Vars

```bash
OKTA_DOMAIN=                # e.g. dev-12345678.okta.com
P3_OKTA_CLIENT_ID=          # Web app client ID (user login)
P3_OKTA_CLIENT_SECRET=      # Client secret (server-side code exchange)
P3_OKTA_AI_AGENT_ID=        # wlp_xxx Workload Principal ID (JWT iss/sub)
P3_OKTA_PRIVATE_KEY=        # JWK JSON string (single line) — signs client assertions

# Shared MCP server vars (also used by P1 and P6)
HR_AUTHZ_SERVER_ID=         # e.g. ausy1ypcn2tp4RtRG1d7
HR_RESOURCE_AUDIENCE=       # e.g. api:hr
FINANCE_AUTHZ_SERVER_ID=
FINANCE_RESOURCE_AUDIENCE=  # e.g. api:finance
```

## Okta Prerequisites

All provisioned automatically by `node scripts/setup.js`:

- **AI Agent (Workload Principal)** — `P3_OKTA_AI_AGENT_ID`; RSA public key registered on the WLP
- **Shared web app** (`okta-demo-p3-p4-shared`) — `client_secret_basic`; redirect URI: `{NEXTAUTH_URL}/api/auth/callback/p3`
- **HR auth server** — XAA (`jwt-bearer`) grant enabled; scopes `hr:read`, `hr:write`, `hr:delete` defined
- **Finance auth server** — XAA (`jwt-bearer`) grant enabled; scopes `finance:read`, `finance:write`, `finance:approve` defined

```bash
docker compose --profile p3 up --build
# → http://localhost:3020/patterns/p3
# Click "Login with Okta", then try:
```

**Suggested demo flow:**
1. *"Show me the org chart for Engineering"* — `get_org_chart` → `hr:read` token
2. *"What's the Q3 budget for Engineering?"* — `get_budget` → `finance:read` token
3. *"Promote Sarah Johnson to Staff Engineer"* — `update_employee_title` → `hr:write` token (different scope, new XAA)

Watch the event stream: each XAA exchange emits events showing the scope requested and the resulting `sub`/`act` claims.
