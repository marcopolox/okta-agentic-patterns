# P4 — Outbound SaaS via Okta STS

**Status:** ✅ Built  
**Auth flow:** User login (PKCE) → id_token → Okta STS token exchange → GitHub/Slack access token → SaaS API  
**Right panel:** Chat

---

## The Story

*"You log into the company portal and ask the AI assistant to list your GitHub repos and post a summary to Slack. The agent doesn't have your GitHub credentials or a Slack API key — it goes to Okta, presents your identity, and Okta brokers access to both services. You authorized it once, and the agent can now act on your behalf across multiple SaaS platforms."*

P4 demonstrates **outbound SaaS integration via Okta STS (Security Token Service)**. The agent holds the user's `id_token` and exchanges it for SaaS-specific access tokens using Okta's token brokering capability — no SaaS credentials are stored in the agent or the `.env` file.

---

## Okta STS Token Exchange

The STS exchange swaps a user's `id_token` for a third-party SaaS access token without the user re-authenticating.

**Endpoint:** `POST https://{OKTA_DOMAIN}/oauth2/{OKTA_AUTH_SERVER_ID}/v1/token`

```
grant_type            = urn:ietf:params:oauth:grant-type:token-exchange
requested_token_type  = urn:okta:params:oauth:token-type:oauth-sts
subject_token         = {user's id_token}
subject_token_type    = urn:ietf:params:oauth:token-type:id_token
resource              = {P4_GITHUB_STS_RESOURCE or P4_SLACK_STS_RESOURCE}
                        ← the OIN integration resource ORN (orn:okta:...)
scope                 = {optional — e.g. repo for GitHub}
client_assertion_type = urn:ietf:params:oauth:client-assertion-type:jwt-bearer
client_assertion      = {signed JWT: iss=sub=P4_OKTA_AI_AGENT_ID, aud=tokenEndpoint}
```

Returns: an `access_token` valid for the target SaaS API.

## Tools

### GitHub tools

| Tool | Description |
|------|-------------|
| `list_repos` | List repositories accessible to the user |
| `search_repos` | Search GitHub repositories by query |
| `get_repo` | Get details for a specific repository |

Calls `https://api.github.com` with the STS-issued GitHub token as Bearer.

### Slack tools

| Tool | Description |
|------|-------------|
| `list_channels` | List accessible Slack channels |
| `get_channel_messages` | Retrieve recent messages from a channel |
| `post_message` | Post a message to a Slack channel |

Calls `https://slack.com/api/` with the STS-issued Slack token.

## End-to-End Flow

```
1.  User clicks "Login with Okta" on /patterns/p4
    → /api/auth/start/p4 → Okta PKCE (P3_OKTA_CLIENT_ID shared app)
2.  Okta → /api/auth/callback/p4
    → console exchanges code for tokens
    → id_token stored in HttpOnly cookie p4_id_token
3.  Page reloads → passes userToken to ChatPanel
4.  User asks: "List my GitHub repos"
    → console sends Authorization: Bearer {id_token} to p4-agent
5.  Agent calls exchangeForToken(userIdToken, P4_GITHUB_STS_RESOURCE, "repo")
    → STS exchange → GitHub access token
6.  Agent calls GitHub API with GitHub token → returns repo list
7.  User asks: "Post a summary to #general on Slack"
    → Agent calls exchangeForToken(userIdToken, P4_SLACK_STS_RESOURCE)
    → STS exchange → Slack bot token (xoxb-...)
    → Agent calls Slack API → posts message
```

---

## Slack Bot Token Limitation

> **Important for demo setup:** Slack STS issues a **bot token** (`xoxb-`), not a per-user token. Bot token scopes are fixed at Slack app installation time by the workspace admin. Passing `scope=chat:write` to the STS exchange returns HTTP 200 but the token only carries the scopes the app was installed with.

If `post_message` fails with Slack error `missing_scope`:
- **Not fixable** by force-reauthorization (that only works for per-user `xoxp-` tokens)
- **Fix**: an Okta admin must add `chat:write` + `chat:write.public` to the Slack OIN app's OAuth scopes in Okta Admin → Applications, then reinstall the app in the Slack workspace

The agent handles `missing_scope` gracefully by returning admin instructions instead of crashing.

## Key Env Vars

```bash
OKTA_DOMAIN=                # Your Okta org domain
OKTA_AUTH_SERVER_ID=        # Auth server for STS exchange (default is fine)
P4_OKTA_CLIENT_ID=          # = P3_OKTA_CLIENT_ID (shared web app)
P4_OKTA_CLIENT_SECRET=      # = P3_OKTA_CLIENT_SECRET (shared web app)
P4_OKTA_AI_AGENT_ID=        # wlp_xxx Workload Principal ID (JWT iss/sub)
P4_OKTA_PRIVATE_KEY=        # JWK JSON string (single line)

# OIN resource ORNs — must be set manually after provisioning
P4_GITHUB_STS_RESOURCE=     # e.g. orn:okta:oauth2:...github...
P4_SLACK_STS_RESOURCE=      # e.g. orn:okta:oauth2:...slack...
```

## Okta Prerequisites

### Automatically provisioned (by `node scripts/setup.js`)

- **AI Agent** `P4_OKTA_AI_AGENT_ID` — RSA public key registered
- **Shared web app** (same as P3) — `P4_OKTA_CLIENT_ID` / `P4_OKTA_CLIENT_SECRET`
- **Token-exchange policy** on `OKTA_AUTH_SERVER_ID`

### Manual setup required

**GitHub Enterprise OIN integration:**
1. Okta Admin → Applications → Browse App Catalog → search "GitHub Enterprise"
2. Add the integration
3. Navigate to the app → Security → STS → enable STS integration
4. Copy the resource ORN (starts with `orn:okta:`)
5. Set `P4_GITHUB_STS_RESOURCE=orn:okta:...` in `.env`

**Slack OIN integration:**
1. Okta Admin → Applications → Browse App Catalog → search "Slack"
2. Add the integration
3. Navigate to the app → Security → STS → enable STS
4. Add OAuth scopes: `chat:write`, `chat:write.public`, `channels:read`
5. Install the app in your Slack workspace
6. Copy the resource ORN
7. Set `P4_SLACK_STS_RESOURCE=orn:okta:...` in `.env`

> The setup script prints step-by-step instructions for both integrations if the ORN values are not already set.

## P3 vs P4 — What's Different

| Aspect | P3 | P4 |
|--------|----|----|
| Target | Internal MCP resource servers | External SaaS APIs (GitHub, Slack) |
| Token exchange | XAA / ID-JAG (2-step, `jwt-bearer` grant) | Okta STS (1-step, `token-exchange` grant) |
| Token type requested | `id-jag` → resource token | `oauth-sts` → SaaS token |
| Resource identifier | Auth server URL | OIN resource ORN |
| Resulting token | Internal resource token (`api:hr`) | GitHub/Slack access token |

Both patterns share the same login flow (same Okta web app client) and the same private-key JWT client assertion pattern.
