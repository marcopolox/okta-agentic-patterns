# P2 — 3rd Party Consumer Agent

**Status:** ✅ Built  
**Auth flow:** Consumer agent → Inventory MCP resource; user delegates access via inline PKCE  
**Right panel:** Chat

---

## The Story

*"You're using a third-party shopping assistant app. It can show you a product catalog without any login — but the moment you ask about stock levels or your order status, it tells you it needs permission. You click a link, log in with Okta, and come straight back to your conversation — now the agent has exactly the access it needs."*

P2 demonstrates the **consumer agent authorization pattern**: a 3rd party agent starts with zero permissions and acquires them inline through a delegated PKCE flow, without breaking the conversation. The agent never handles credentials directly — Okta issues a scoped access token that the agent uses as a Bearer token.

---

## Architecture

The P2 agent operates in two modes depending on whether a user token is present:

| Mode | Token | Visible tools | System prompt |
|------|-------|--------------|---------------|
| Public | None | `get_product_catalog` only | Catalog-only; includes auth link |
| Authorized | `access_token` (Bearer) | All 5 Inventory tools | Full inventory access |

This two-mode design is critical: without a token, the agent only exposes the one public tool to the LLM — preventing hallucination or 401 errors on protected calls. With a token, all five tools become available.

---

## End-to-End Flow

```
1.  User visits /patterns/p2 → no token in cookie → public mode
2.  User asks: "What products do you have?"
    → p2-agent connects to Inventory MCP (no auth header)
    → only get_product_catalog visible to LLM
    → agent responds with catalog data
3.  User asks: "Check stock for WirelessPro Headphones"
    → LLM would need check_stock (not visible in public mode)
    → agent replies with an [Authorize this agent](/api/auth/start/p2) link
4.  User clicks the link
    → ChatPanel saves messages + session ID + pending question to sessionStorage
    → full-page redirect to /api/auth/start/p2
5.  Console redirects to Okta PKCE flow
    → auth server: INVENTORY_AUTHZ_SERVER_ID
    → scope: inventory:read
6.  User logs in → Okta redirects to /api/auth/callback/p2
    → console exchanges code for access_token
    → access_token stored in HttpOnly cookie p2_access_token
    → redirect to /patterns/p2
7.  Page reloads as server component
    → reads p2_access_token cookie → passes userToken to ChatPanel
8.  ChatPanel restores messages from sessionStorage
    → detects new userToken → auto-resends the pending question
9.  Console sends Authorization: Bearer {access_token} to p2-agent
10. Agent connects to Inventory MCP with the Bearer token
    → all 5 tools visible → full response
```

---

## Token Anatomy

```
iss  = Inventory auth server (INVENTORY_AUTHZ_SERVER_ID)
sub  = user email
aud  = INVENTORY_RESOURCE_AUDIENCE  (e.g. api://inventory-resource)
scp  = inventory:read
```

## Inventory MCP Server

Five tools, one public:

| Tool | Auth required | Description |
|------|--------------|-------------|
| `get_product_catalog` | None (public) | List products; optional `category` filter |
| `check_stock` | `inventory:read` | Stock level for a SKU: `inStock`, `reserved`, `available`, `status` |
| `get_product_details` | `inventory:read` | Full product details + live stock for one SKU |
| `get_order_status` | `inventory:read` | Order status, ETA, tracking number |
| `list_categories` | `inventory:read` | All categories with product counts |

**Auth guard design**: MCP handshake messages (`initialize`, `notifications/initialized`, `tools/list`) pass through without auth. Only `tools/call` for non-public tools requires a valid Bearer token. This allows the agent to connect and discover tools before the user is authorized — the 401 only fires at call time if the user attempts a protected operation without a token.

**Mock data**: 6 consumer electronics products (WirelessPro Headphones X3, SmartWatch Ultra 2, 4K Webcam Pro, Mechanical Keyboard TKL, Portable SSD 1TB, USB-C Hub 7-in-1). 3 sample orders (ORD-10041, ORD-10042, ORD-10043).

The server also exposes `/.well-known/oauth-protected-resource` returning the Inventory auth server URL and supported scopes.

## Session Preservation

The inline PKCE redirect would normally wipe the chat conversation. P2 uses `sessionStorage` to survive the redirect:

1. Before redirect, `ChatPanel` (with `preserveSessionOnNavigation` prop) saves messages, session ID, and the pending unanswered question to `sessionStorage`
2. After redirect back to `/patterns/p2`, `ChatPanel` restores messages from `sessionStorage` on mount
3. When the new `userToken` prop arrives, `ChatPanel` auto-resends the pending question

The user sees no conversation loss — the authorize/callback/resend cycle is invisible.

## Key Env Vars

```bash
P2_OKTA_CLIENT_ID=            # Web app client ID (PKCE flow in console)
P2_OKTA_CLIENT_SECRET=        # Client secret for server-side code exchange
INVENTORY_AUTHZ_SERVER_ID=    # Okta auth server ID for Inventory
INVENTORY_RESOURCE_AUDIENCE=  # Token audience, e.g. api://inventory-resource
```

## Okta Prerequisites

- Web app OAuth client with PKCE enabled; redirect URI: `{NEXTAUTH_URL}/api/auth/callback/p2`
- Dedicated Okta authorization server for Inventory with `inventory:read` scope defined
- User accounts in the Okta org for demo logins

All of the above are provisioned automatically by `node scripts/setup.js`.
