# P5 — Human-in-the-Loop

**Status:** 🔲 Pending  
**Auth flow:** User login (PKCE) → id_token → XAA → resource token → tool call gated by human approval  
**Right panel:** Chat

---

## The Story

*"The agent is about to promote an employee, post to a public Slack channel, or approve a budget change. It pauses, shows you exactly what it's about to do, and waits for a thumbs-up before proceeding. If you reject, the agent explains what it would have done and stops. You stayed in control."*

P5 demonstrates the **human-in-the-loop** pattern: an agent that identifies high-risk or irreversible actions and suspends its own agentic loop to request explicit human approval before proceeding. The agent acts on behalf of the user (same XAA flow as P3) — the approval gate is an application-level control, not an OAuth flow.

---

## Intended Flow

```
1.  User logs in via console (same PKCE flow as P3)
    → id_token stored in HttpOnly cookie
2.  User sends prompt: "Promote Sarah Johnson to Staff Engineer"
3.  Agent identifies this triggers update_employee_title (hr:write)
    → classified as a "sensitive" / "irreversible" action
4.  Agent suspends loop; formats a human-readable approval request:
      Action:   update_employee_title
      Employee: Sarah Johnson (ID: emp_042)
      Change:   "Senior Engineer" → "Staff Engineer"
      Risk:     Irreversible without manual HR system correction
5.  Agent returns approval-request message to chat panel
6.  User sees an Approve / Reject UI in the chat
7a. User clicks Approve
      → agent resumes; performs XAA (hr:write scope)
      → calls update_employee_title → reports outcome
7b. User clicks Reject
      → agent explains what it would have done; does not execute
      → loop terminates cleanly
```

---

## Approval Gate Design

The approval gate is application-layer — the OAuth/XAA flow is identical to P3. The gate sits between the LLM's decision to call a tool and the actual tool execution:

```
LLM decides to call tool
  ↓
Is tool in REQUIRES_APPROVAL set?
  ↓ yes                          ↓ no
Return approval-request        Execute immediately
message to console               (same as P3)
  ↓
User approves / rejects
  ↓ approve                      ↓ reject
Resume loop;                   Return explanation;
execute tool                   abort loop
```

**Sensitive tools (planned):**
- `update_employee_title` — modifies permanent HR record
- Any tool with `hr:write` or `finance:write` / `finance:approve` scope

## Auth Note

P5 uses the same login as P3 (`requiresUserToken: true`). The agent would perform standard XAA to obtain narrow-scoped resource tokens — the approval step is purely an agent-side control. No new Okta resources are required beyond what P3 already provisions.
