# P1 — 3rd Party Coding Assistant

**Status:** ✅ Built  
**Auth flow:** 3rd party coding assistant → Okta MCP Bridge → HR + Finance MCP resource servers  
**Right panel:** Connection guide (step-by-step setup instructions)

---

## The Story

*"You're using Claude Code and you ask it to look up an employee. Claude has no idea your company uses Okta for HR — but when connected to the Okta bridge, it discovers all available tools automatically. Each time it calls a tool, the bridge handles the entire XAA flow: it acquires a scoped token from the right auth server and forwards the call to the correct MCP resource server on the operator's behalf."*

P1 demonstrates a **3rd party coding assistant** — a third-party AI tool (Claude Code, Cursor, VS Code Copilot) gaining access to protected enterprise resources through a single bridge endpoint, with Okta handling all identity and authorization transparently.

---

## What Runs

- `hr-server` (port 3101) — HR MCP resource server
- `finance-server` (port 3102) — Finance MCP resource server
- Console pattern page at `/patterns/p1` shows the `ConnectionGuide` component

There is **no P1 agent** in this repo. The operator's coding assistant IS the agent. The Okta MCP Bridge is an external pre-built product — it runs on the host, not in Docker.

## End-to-End Flow

```
1. Operator starts the Okta MCP Bridge on the host (port 8008)
2. Operator adds the bridge as an MCP server in their coding assistant:
     Claude Code:  claude mcp add --protocol http --name okta-demo --url http://localhost:8008
     VS Code:      .vscode/mcp.json → { "servers": { "okta-demo": { "type": "http", "url": "..." } } }
     Cursor:       Settings → MCP → Add Server → HTTP → http://localhost:8008
3. Coding assistant connects → receives 401 → performs OAuth discovery:
     a. Fetches /.well-known/oauth-protected-resource → gets bridge as authorization server
     b. Fetches /.well-known/oauth-authorization-server → gets authorization_endpoint,
        token_endpoint, and registration_endpoint
     c. Calls POST /.well-known/oauth/registration (RFC 7591 DCR) → bridge generates
        a fresh adapter-scoped client ID: dcr_<16-char-hex> (e.g. dcr_3f8a21b09c4e1d7f)
4. Coding assistant uses dcr_xxx client ID to initiate PKCE → operator logs in with Okta
5. Bridge intercepts the token exchange (BFF pattern):
     - Returns access_token to the coding assistant
     - Stores access_token → id_token mapping internally (keyed by SHA-256(access_token))
6. Coding assistant sends tools/list → bridge returns aggregated HR + Finance tool list
7. Operator asks: "How many employees are in Engineering?"
8. Assistant calls list_employees with Authorization: Bearer <access_token>
9. Bridge looks up the stored id_token, performs XAA ID-JAG exchange to get a
   scoped resource token (hr:read), forwards the tool call to hr-server
10. hr-server validates the token via JWKS, executes the tool, returns data
11. Console event stream shows live XAA + tool-call events in real time
```

---

## Token Anatomy

```
iss  = HR or Finance auth server (HR_AUTHZ_SERVER_ID / FINANCE_AUTHZ_SERVER_ID)
sub  = operator's user identity (delegated through the adapter)
aud  = api:hr  or  api:finance
scp  = hr:read / finance:read  (per tool call)
```

The `act` claim shows the agent (bridge) acting on behalf of the operator (`sub`), making the full delegation chain visible in the event stream.

---

## How the Bridge Handles Tokens (access_token vs id_token)

When the coding assistant completes the PKCE login at step 3, Okta returns **both** an `access_token` and an `id_token`. These serve different roles:

| Token | Who holds it | What it's used for |
|---|---|---|
| `access_token` | Coding assistant | Sent as `Authorization: Bearer` on every request to the bridge |
| `id_token` | Bridge only (never given to the coding assistant) | Used as `subject_token` in the XAA ID-JAG exchange with Okta |

**The bridge's internal flow on every tool call:**
1. Receives `Authorization: Bearer <access_token>` from the coding assistant
2. Computes `SHA-256(access_token)` → looks up the stored `id_token` in `UserTokenStore`
3. Uses that `id_token` as the `subject_token` for XAA ID-JAG step 1 (exchange for an ID-JAG token)
4. Exchanges the ID-JAG for a scoped resource token at the HR or Finance auth server
5. Forwards the tool call with the scoped `Bearer` token to the MCP resource server

The coding assistant has no knowledge of the `id_token` or the XAA flow — it simply holds an `access_token` and uses it like any other bearer credential. The entire identity delegation machinery runs transparently inside the bridge.

## Key Env Vars

```bash
# MCP resource servers (shared with P3, P6)
HR_AUTHZ_SERVER_ID=         # Okta auth server ID for HR
HR_RESOURCE_AUDIENCE=       # e.g. api:hr
FINANCE_AUTHZ_SERVER_ID=    # Okta auth server ID for Finance
FINANCE_RESOURCE_AUDIENCE=  # e.g. api:finance

# Bridge URL — shown in the ConnectionGuide so operators know where to point their tool
MCP_ADAPTER_URL=http://localhost:8008
```

## Okta Prerequisites

- HR and Finance authorization servers provisioned (handled by `node scripts/setup.js`)
- XAA (jwt-bearer) grant enabled on both servers for the bridge's client
- Operator has a user account in the Okta org (for PKCE login through the bridge)

## How to Run

```bash
# The standalone Okta MCP Bridge must be running on the host first.
# Configure it with your Okta credentials and backend MCP server URLs:
#   HR:      http://host.docker.internal:3101
#   Finance: http://host.docker.internal:3102

docker compose --profile p1 up --build
# → http://localhost:3020/patterns/p1
# Follow the ConnectionGuide on the pattern page.
```

---

## Important Note on the Bridge

The `okta-agent-mcp-adapter` is an **external Okta product** shipped as a pre-built Docker image. This repo only provides configuration for it. Never attempt to modify the bridge's source code — it lives entirely outside this repository.

`MCP_ADAPTER_URL` in `.env` controls the URL displayed in the `ConnectionGuide` component. Change it if you're hosting the bridge at a different address.
